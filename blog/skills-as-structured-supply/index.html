Skills as Structured Supply

How to give the supply tier form — and why form is the difference between a docs folder and a pull system.

The Gap in the Framework

The thesis describes three tiers. The constitution is crisp — 30 to 50 lines, hard stops, always loaded. The manifest is crisp — 10 to 15 lines, routing table, always visible. The supply tier is described by what it holds: focused documents, pulled on demand, organized by retrieval key.

That is necessary but not sufficient. It tells you what the supply contains. It does not tell you what the supply looks like.

In practice, a supply tier without form degrades into a docs folder. Someone writes a markdown file about network standards. Someone else writes one about deployment procedures. A third person writes an architecture decision record. Each document has a different structure, a different level of detail, a different idea of what "focused" means. The manifest routes to them. The agent pulls them. And the quality of what arrives varies wildly — because the supply has content but no contract.

Skills are the contract.

What a Skill Is

A skill is a supply document with a predictable shape. It has three parts:

Trigger. When does this skill apply? The trigger is a description that maps to task patterns — the same patterns the manifest routes on. "Use when creating or editing Word documents." "Use when modifying database migration files." The trigger is what makes manifest routing mechanical instead of ad hoc. The agent reads the manifest, matches a trigger, pulls the skill. No ambiguity about which supply document applies.

Scope. What does the agent need to know to do the work? This is the invisible knowledge — the instructions, constraints, best practices, and hard-won lessons that the agent cannot infer from the code. A skill is scoped to one concern. Not "everything about the database" but "how we write and validate migration files." Tight enough that the entire skill is relevant when pulled. Tight enough that loading it is cheap.

Boundary. Where does this skill end? What is explicitly out of scope? What should the agent not attempt, and where should it hand off? The boundary prevents a skill from becoming a junk drawer. It also prevents the agent from over-applying a skill's instructions to adjacent work that needs different context.

Here is what the contract looks like in practice:

# Skill: Database Migrations

## Trigger
Use when creating, modifying, or reviewing database migration files.

## Scope
- Migrations use Alembic with async SQLAlchemy.
- Every migration must include a rollback path. No exceptions.
- Column renames require a two-phase migration: add new, backfill,
  drop old. Single-step renames break running deployments.
- We hit a production outage in 2024 from a migration that locked
  a 200M-row table. All ALTER TABLE on large tables must use
  pt-online-schema-change or equivalent. The ORM will not protect you.
- Test migrations against a snapshot of production data before merge.
  The CI job is: make test-migrations

## Boundary
- Schema design decisions are out of scope. See: architecture-decisions/
- ORM model changes that do not involve migrations do not need this skill.
- Performance tuning of existing queries is a separate concern.

That is the contract: trigger, scope, boundary. Every skill follows it. The agent knows what to expect when it pulls one. The human knows what to write when creating one. The scope holds invisible knowledge — the outage, the two-phase rename requirement, the CI job name — things no amount of code reading reveals.

Where Skills Fit

Skills live inside the supply tier. They are not a fourth tier. They are the supply tier with form.

Not all supply is skills. Architecture decision records, vendor reference material, compatibility notes — these do not follow the skill contract. They are pulled on demand but not structured as trigger-scope-boundary units. That is fine. But the supply that agents pull repeatedly, the supply that directly governs how work gets done — that supply benefits from a predictable shape.

Over time, the ratio shifts. The reference doc that kept getting pulled gets rewritten as a skill. The ADR that informed every auth change gets distilled into a skill for auth modifications. The supply tier does not start as all skills. It evolves toward skills as the human calibrates what the agent actually needs — not what they guessed it would need, but what it actually needed after ten sessions, after fifty.

One Skill, One Concern

One skill covers one concern. Not one topic — one concern. The difference matters. "The database" is a topic. "How we write and validate migration files" is a concern. "Authentication" is a topic. "How we handle token refresh in the mobile client" is a concern.

The test: if the skill is pulled, is all of it relevant? If half the skill is about something the current task does not need, it is too broad. Split it. The workbench rule applies inside the supply tier, not just at the tier boundary.

Skills also follow the same lifecycle pressures as the constitution. They will bloat. Every near-miss adds a paragraph. The fix is the same: decomposition. When a skill exceeds its useful size, one concern became two. Split it, update the manifest, and both skills are lean again. Skills go stale too — the library was replaced, the constraint was removed. A stale skill loads confidently wrong context, which is worse than no skill at all.

Harness Reality

The standard says skills live in .context/skills/. That is the canonical path. It is also a path that no agent harness reads from natively today. Every harness has its own opinion about where context files live, how they load, and whether conditional loading is supported.

This is the honest landscape as of early 2026:

Claude Code. Reads CLAUDE.md as its instruction file. The agent can read files from a .claude/ directory when instructed to. Skills work here — but the pull mechanism is human-engineered, not harness-native. The human writes routing logic in CLAUDE.md that tells the agent which files to load for which tasks. The agent follows those instructions. This is the closest to the standard's model, but the harness is not doing the routing — the human is, through the constitution.

Cursor. Has moved past the single .cursorrules file (now deprecated) to a .cursor/rules/ directory with .mdc files. Each rule file supports metadata: glob patterns for file matching, an alwaysApply flag, a description for agent-requested loading, and manual invocation. This is structurally close to conditional loading — rules attach when matching files are referenced. The mechanism is different from the standard's manifest routing, but the principle is converging: context loads based on what the task touches, not by default.

Windsurf. Similarly evolved from .windsurfrules to a .windsurf/rules/ directory with activation modes: always on, manual, glob-based, or model decision. Like Cursor, the harness is moving toward conditional context — rules load based on relevance, not by default.

GitHub Copilot. Reads AGENTS.md. No directory-based context loading. No conditional activation. Everything the agent sees must be in the instruction file or discovered through code navigation. This is the most push-oriented harness in common use today.

The trajectory is clear. Harnesses are converging on directory-based, conditionally-loaded context — which is the pull model under different names. Cursor calls them "rules" with glob patterns. Windsurf calls them "rules" with activation modes. The standard calls them "skills" with triggers. The mechanism varies. The direction is the same: stop loading everything at session start.

But none of them read from .context/. Each harness defines its own path.

Bridging the Gap

The standard already solves this for constitutions. CONSTITUTION.md is the source of truth. Symlinks bridge to harness-specific filenames:

ln -s CONSTITUTION.md CLAUDE.md
ln -s CONSTITUTION.md AGENTS.md

The same pattern extends to skills, but the bridge depends on what the harness supports. For harnesses with directory-based loading, the canonical .context/skills/ can be symlinked or mirrored to the harness-specific path. For push-only harnesses, the bridge is the manifest itself — embedded in the instruction file, telling the agent which supply documents exist and where to find them.

your-repo/
├── CONSTITUTION.md                ← canonical source of truth
├── MANIFEST.md                    ← canonical routing table
│
├── .context/skills/               ← canonical skill location
│   ├── db-migrations.md
│   ├── auth-flow.md
│   └── deploy-staging.md
│
├── CLAUDE.md → CONSTITUTION.md    ← symlink for Claude Code
├── AGENTS.md → CONSTITUTION.md    ← symlink for Copilot
│
├── .claude/
│   └── skills → ../.context/skills/
│
├── .cursor/rules/                 ← harness-native format
│   └── db-migrations.mdc         ← can reference .context/ skills
│
└── .windsurf/rules/               ← harness-native format
    └── db-migrations.md           ← can reference .context/ skills

The principle: author skills once in the canonical location. Bridge to harness-specific paths. The skill contract — trigger, scope, boundary — is the same regardless of the path the harness reads from. When a harness adds features, the bridge simplifies. When it does not, the bridge compensates.

For Cursor and Windsurf, the bridge might be thin — their rule files can reference canonical skill files using file includes or path references, keeping the harness-specific file as a shim that points to the source of truth. The skill content lives in .context/skills/. The harness file provides the activation metadata in the format the harness expects.

The Standard Is Not the Harness

This distinction matters. The Lean Context Standard defines the architecture: constitution, manifest, supply. Skills define the form of structured supply: trigger, scope, boundary. These are harness-agnostic. They describe how to organize context for any agent, regardless of which tool loads it.

The harness is the delivery mechanism. It determines where files live, how they load, whether conditional loading is native or engineered. The harness will change. New tools will appear. Existing tools will add features — as Cursor and Windsurf already have. The standard should not be coupled to any of them.

What the standard provides is the contract that survives the harness. When a team switches from one agent tool to another, the constitution does not get rewritten. The skills do not get restructured. The manifest does not change. Only the bridge changes — which filenames are symlinked, which paths are mirrored, which metadata format wraps the content. The architecture is portable. The delivery adapts.

If the standard were defined in terms of .claude/, it would be a Claude standard. If defined in terms of .cursor/rules/, a Cursor standard. By defining it as .context/ with bridges, it belongs to the practitioner — not the vendor.

Form Enables Function

Without form, every supply document is a snowflake. The human writes what they think the agent needs, in whatever structure feels natural. The agent pulls it and parses whatever it gets. Sometimes it works. Sometimes the document is too broad, or too vague, or structured in a way that buries the relevant instruction under three paragraphs of background.

With form — trigger, scope, boundary — every supply document follows a contract. The human knows what to write. The agent knows what to expect. The manifest routing is mechanical. The pull rule works because what gets pulled is reliably useful.

Form is not bureaucracy. It is the difference between a parts bin and a supply chain. Both hold inventory. Only one delivers the right part at the right time.

Summary

The supply tier needs form, not just content. Skills are supply documents with a contract: trigger, scope, boundary. The trigger makes manifest routing mechanical. The scope delivers invisible knowledge for one concern. The boundary prevents bloat and over-application.

Skills live canonically in .context/skills/. The harness determines the delivery path. Harnesses are converging on conditional loading — directory-based context that activates by relevance, not by default. The standard names the contract. The bridge adapts it to each harness. When the harness changes, the bridge changes. The skills stay.

The standard is not the harness. The architecture survives the tooling.
