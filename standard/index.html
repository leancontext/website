The Lean Context Standard

A Structural Specification for Human-Agent Repositories

What This Is

This is the implementation spec for the Lean Context Thesis. The thesis is the philosophy — why curated, modular, pull-based context makes humans and agents smarter together. This document is the structure — how you organize a repository so that philosophy is enforceable, not aspirational.

This spec is agent-agnostic and model-agnostic. It does not care whether you use Claude, GPT, Gemini, or a local model. It does not care whether your agent runs through Claude Code, Cursor, Aider, OpenCode, or a custom harness. It cares about the cultural contract between the human and the agent, and the repository structure that encodes it.

The Cultural Contract

The Lean Context standard is a two-sided agreement.

The human commits to:

  • Keeping the constitution under budget and load-bearing.
  • Maintaining the manifest as a live routing table.
  • Writing invisible knowledge into supply documents when the agent surfaces gaps.
  • Not documenting what the agent can infer from the codebase.
  • Running compaction periodically — demoting, pruning, validating.
  • Ending sessions when the task is complete. One task, one session, clean workbench. The session boundary is the primary compaction mechanism.
  • Trusting the agent to pull what it needs.

The agent commits to:

  • Reading the constitution before any work begins.
  • Consulting the manifest to identify relevant supply documents.
  • Pulling supply documents when the task matches a route.
  • Stopping when operating without sufficient grounding — the andon rule. Surfacing what it thinks it is missing rather than guessing.
  • When a gap is surfaced, helping the human fill it — articulating what is needed, collaborating on the supply document, confirming the result would have been sufficient for the task.
  • Never treating the constitution as optional or the manifest as decoration.

This is not a technical integration. It is a working agreement. The repository structure makes the agreement enforceable by giving each commitment a specific location, a loading strategy, and a failure mode the human can diagnose.

Repository Structure

<repo-root>/
├── CONSTITUTION.md          ← always loaded, hard stops only
├── MANIFEST.md              ← always visible, routing table
├── .context/                ← supply documents, pulled on demand
│   ├── <topic>.md
│   ├── <topic>.md
│   └── ...
└── <your code>

The names are deliberate. CONSTITUTION.md and MANIFEST.md are uppercase because they are governance documents, not implementation details. They sit at the repository root because they apply to all work in the repository. The .context/ directory is dotfile-prefixed because its contents are infrastructure — available to the agent, invisible in casual directory listings, and not part of the application.

Adapting to tool conventions. Some agent tools expect specific file names or locations. Claude Code loads CLAUDE.md automatically. Cursor reads .cursorrules. Other tools have their own conventions. The standard accommodates this: use whatever filename your tool requires for the constitution, and symlink or include from the canonical location if needed. The principle — not the filename — is what matters. A constitution named CLAUDE.md is still a constitution. A manifest embedded in a .cursorrules header is still a manifest. Adapt the names. Keep the structure.

The Constitution

Purpose. The constitution contains rules that, if missed once, cause damage that survives past the session. It is always loaded — every session, every task, no exceptions.

What belongs here:

  • Hard stops. “Never run destructive commands without confirmation.”
  • Security boundaries. “Never expose credentials in code or logs.”
  • Data integrity rules. “Never write directly to production databases.”
  • Non-negotiable procedures. “Always run the test suite before committing.”
  • The andon instruction. “When you lack sufficient context to proceed with confidence, stop and surface what you think you are missing. Do not guess.”

What does not belong here:

  • Style guides. The agent can infer these from the codebase, or they belong in linters.
  • Architectural overviews. These are supply documents.
  • Workflow preferences. These are supply documents.
  • Lessons learned. These belong in the supply, not the constitution. The constitution is not a journal.
  • Anything the agent could figure out by reading the code.

The test. For every line in the constitution, ask: if the agent misses this once, is the damage irreversible? “Irreversible” means damage that survives past the session — corrupted production data, a security boundary crossed, a destructive command executed without confirmation. If the mistake can be caught in code review or rolled back from version control, it is supply material, not a constitutional rule.

Budget. 30 to 50 lines for most projects. If the constitution exceeds 100 lines, it contains reference material masquerading as rules. Run compaction.

Format. Plain text or markdown. No frontmatter, no metadata, no structured headers beyond what aids readability. The constitution is read by the agent as raw context — every token counts. Write it the way you would write a checklist for a colleague who is about to operate on production: short, direct, unambiguous.

The Manifest

Purpose. The manifest is a routing table that maps task patterns to supply documents. It tells the agent what context is available and when to load it — without holding any of that context itself.

What belongs here:

  • Task-to-document mappings. “If modifying network configuration, load .context/network-standards.md.”
  • Trigger conditions. “If working with authentication, load .context/auth-architecture.md.”
  • Scope indicators. “If the task involves Ceph storage, load .context/ceph-operations.md and .context/storage-decisions.md.”

What does not belong here:

  • The context itself. The manifest never contains reference material, procedures, or architectural knowledge. It contains pointers.
  • Rules or prohibitions. Those belong in the constitution.
  • General project descriptions. The agent can infer these.

Format. The manifest is a short, scannable document. Each entry is a trigger condition and a path. The agent reads the manifest at session start, matches against the current task, and pulls the relevant documents.

Example:

# Manifest

- Networking tasks → .context/network-standards.md
- Authentication or identity → .context/auth-architecture.md
- Deployment or release → .context/deploy-procedures.md
- Ceph or storage operations → .context/ceph-operations.md, .context/storage-decisions.md
- Database schema changes → .context/database-conventions.md
- CI/CD pipeline changes → .context/ci-pipeline.md
- New service creation → .context/service-template.md

Budget. 10 to 15 lines. If the manifest exceeds 25 entries, the supply is either too granular (merge related documents) or the project needs sub-manifests per domain.

The Supply

Purpose. Supply documents contain the invisible knowledge that enables depth. They are pulled on demand — loaded when the task requires them, never pushed in bulk.

What belongs here:

  • Architectural decisions and the reasoning behind them.
  • Operational procedures that are not obvious from the code.
  • Historical context — what was tried, what failed, and why.
  • External constraints that the code does not encode.
  • Domain-specific reference material needed for certain task types.

What does not belong here:

  • Inferable knowledge. File layouts, code conventions, dependency relationships, API patterns — the agent will discover these. Do not spend human effort maintaining documents the agent does not need.
  • Anything that belongs in the constitution. If it prevents irreversible damage, promote it.

Format. Each supply document covers one topic. It is small enough that loading it is cheap (aim for under 200 lines) and specific enough that all of it is relevant to the task that triggered the pull.

Naming. Name supply documents by retrieval key — the topic the agent is looking for — not by author, date, or ticket number.

Good:  network-standards.md, auth-architecture.md, ceph-operations.md
Bad:   jay-notes-march.md, HDI-805-decisions.md, meeting-2026-03-15.md

Staleness. Every supply document should carry a last-reviewed date in its header or frontmatter. Documents untouched beyond a threshold (project-dependent, but 90 days is a reasonable default) should be flagged for review. Stale supply documents are worse than missing ones — they give the agent confidence in outdated information.

The Knowledge Boundary

Not all knowledge belongs in the system. The decision framework:

Inferable knowledge — do not document.

The agent can discover this from the codebase. Documenting it creates maintenance burden for the human and adds noise for the agent. Examples:

  • File and directory layout
  • Code conventions and style patterns
  • Dependency relationships
  • API shapes and signatures
  • Import structures
  • Test file locations and naming patterns

If the agent can learn it by reading the code, it does not earn a place in the system.

One exception: if rediscovery is expensive (large monorepo, complex build system), the system — not the human — can generate a lightweight summary (repo map, dependency graph, recent changelog) and make it available for pull. The rule is: the human does not write or maintain inferable knowledge. Automation can.

Invisible knowledge — document it.

The agent cannot discover this from the codebase. It lives only in the human’s head until it is written down. Examples:

  • Why a decision was made (not what the decision is — the code shows that).
  • What was tried and rejected, and why it failed.
  • External constraints the code does not encode (regulatory, vendor, organizational).
  • Non-obvious tool choices (“we use this tool because the obvious alternative has a specific failure mode in our environment”).
  • Consequences of actions that are invisible until it is too late (“this command destroys data that cannot be recovered”).

If the agent could work on this codebase for a year and never discover it, it is invisible knowledge. Write it down.

The human’s job. Stop documenting what the agent can see for itself. Spend that time writing down the things it never could. The knowledge boundary is not about reducing documentation — it is about redirecting human effort to where it is irreplaceable.

The Andon Rule

The andon rule is the most important behavioral instruction in the constitution. It is also the one most likely to be ignored by default, because it fights the agent’s training.

Language models are next-token predictors trained to be helpful. Their default behavior is to produce an answer — any answer — rather than stop and say they cannot. The andon rule overrides this default.

The instruction. Include this (or your own version of it) in the constitution:

When you lack sufficient context to proceed with confidence, stop.
Do not guess. Do not fill gaps with assumptions. State what you
believe you are missing and ask. The act of stopping is the system
working, not failing.

What the agent should do when it pulls the andon cord:

  • Name the specific uncertainty. “I am modifying the authentication flow but I do not see how token refresh is handled. Is there a supply document for this?”
  • State what it would assume if forced to continue. “Without more context, I would assume refresh tokens are stored in the session, but I am not confident.”
  • Wait for the human to provide context, point to a supply document, or confirm the assumption.

What the human should do when the agent pulls the andon cord:

  • If the answer exists in a supply document the agent missed: point to it. Consider whether the manifest route is clear enough.
  • If the answer is invisible knowledge that has not been documented: work with the agent to write the supply document. The agent helps articulate what is missing, the human provides the invisible knowledge, and together they produce a document that would have been sufficient for the task that triggered the stop.
  • If the agent’s assumption is correct: confirm it and consider whether the knowledge should be documented for next time.

After the gap is filled: add the manifest route. The supply document is now routable. The session ends. The next session — or a fresh retry of the original task — starts with the gap filled and the system improved. Every andon stop that produces a supply document is the system getting smarter.

Alert fatigue. In the early stages, the agent will stop often. The supply is thin. Common questions have not been documented. This friction is temporary. As the supply matures, the agent stops less — not because it is guessing more, but because the system has fewer gaps. Patience in the early reps is the price of a system that works in the later ones.

Session Discipline

The most important compaction mechanism is not a process — it is a habit. When the task is done, the session ends.

Every agent harness has some form of runtime compaction — summarizing conversation history, truncating old tool outputs, offloading content to disk when the context window fills up. This is reactive. It happens because the session has been alive too long and the window is running out of space. The agent is now reasoning through a compressed, lossy summary of work it actually did. The quality of its output degrades in ways the human cannot easily diagnose.

The Lean Context standard makes runtime compaction unnecessary by eliminating the conditions that require it.

A well-structured session starts with a clean workbench: the constitution (small), the manifest (tiny), and one or two supply documents pulled for the specific task. The agent does the work. The work finishes. The session ends. The next task starts a new session with a fresh workbench. The agent never reaches 75% of its context window because the session does not live long enough to accumulate that much state.

This is the lean manufacturing principle fully applied. The workbench is cleared between operations. Inventory does not accumulate during the shift. Each operation starts clean, and ending the session is the clearing action.

The discipline: do not keep sessions alive across tasks. Do not treat a session as a persistent workspace where the agent accumulates context over hours. A session is a unit of work. It ends when the work is done.

There are two clean endings:

  1. The problem is solved. The bug is fixed, the feature is implemented, the document is written. The session produced a code change. It ends.

  2. A gap was discovered and filled. The agent pulled the andon cord. The human and agent collaborated to write a supply document. The manifest route was added. The session produced a context change. It ends. The next session — or a fresh retry of the original task — starts with the gap filled.

Every session leaves the repository better than it found it. Either the code improved or the context system improved. Nothing lingers. If there is a follow-up task, it gets a new session with its own clean workbench and its own targeted supply pulls.

This is uncomfortable for humans who are used to long-running conversations with AI. But the thesis is clear: the agent reasons better with a clean workbench than with the accumulated residue of prior tasks. Ending the session is not losing context. The context lives in the repository — in the constitution, the manifest, and the supply. It persists across sessions because it is structure, not conversation history.

When a long session is unavoidable: some tasks are genuinely large — a multi-file refactor, a complex debugging session that spans multiple subsystems. In these cases, the session will grow. The mitigation is the same principle applied at a smaller scale: when a subtask within the session is complete and the agent is pivoting to a different concern, the human can start a new session. The question is always: is the agent still reasoning about the current problem, or is it dragging the weight of the last problem into the next one?

Context Compaction

There are two completely different operations that can be called “compaction.” They should not be confused.

Runtime compaction is what agent harnesses do mid-session when the context window fills up. Summarizing old messages, truncating tool outputs, offloading content to disk. This is a mechanical, reactive operation. It trades fidelity for space. The agent loses access to the original content and works from a compressed approximation. This is a symptom of a session that has lived too long or loaded too much.

Lean compaction is what the human does to the repository structure between sessions. Demoting constitution entries to the supply. Pruning stale supply documents. Validating manifest routes. This is an intentional, proactive operation. It keeps the system lean so that sessions start clean and stay clean.

The relationship between the two: if you do lean compaction well, the agent should rarely or never need runtime compaction. The constitution is small. The manifest is tiny. Supply documents are loaded only when earned. Sessions end when tasks end. The context window never fills because the system was designed so it would not have to.

Runtime compaction is the agent’s safety net. Lean compaction is the human’s discipline. The standard is built around the discipline, not the safety net.

Constitution compaction.

Review every line. For each rule, ask: if the agent misses this once, is the damage irreversible? If the answer is no, demote the line to a supply document. It does not get deleted — it becomes reference material, available when relevant, not loaded by default.

Watch for:

  • Lessons learned masquerading as hard stops.
  • Style preferences that crept in after an incident.
  • Workflow reminders that belong in the manifest or supply.
  • Rules that were added in fear and never re-evaluated.

The constitution is where rules go to be sacred. Sacred is the enemy of lean.

Manifest compaction.

Review every route. For each entry, ask:

  • Does this supply document still exist?
  • Is the trigger condition still accurate? Do tasks actually match this pattern?
  • Has the supply document it points to been reviewed recently?
  • Are there new supply documents that need routes?

Dead routes are silent failures — the manifest says context is available, but the agent pulls nothing or pulls stale material.

Supply compaction.

Review every document. For each, ask:

  • Is this still accurate?
  • Has the codebase changed in ways that invalidate this context?
  • Is this document ever actually pulled? (If your tooling supports access logs, check them.)
  • Can this be merged with another document that covers related territory?
  • Is any of this now inferable from the codebase? If the code has been refactored to make a decision obvious, the supply document explaining that decision may no longer be needed.

Cadence. Run constitution compaction every time you add a rule — the addition is the trigger to review what is already there. Run manifest and supply compaction on a regular schedule. Monthly is a reasonable starting point. Adjust based on how fast your codebase and team are changing.

Automation. Automate what you can:

  • Line count checks on the constitution (fail CI if over budget).
  • Staleness flags on supply documents (alert if last-reviewed exceeds threshold).
  • Dead route detection in the manifest (check that referenced files exist).
  • Orphan detection in supply (flag documents not referenced by any manifest entry).

Automation enforces structure. Human judgment does the curation.

Getting Started

For a new project:

  1. Create CONSTITUTION.md at the repo root. Write the hard stops you already know — the things that, if the agent got wrong, would cause real damage. Include the andon instruction. Keep it under 50 lines.
  2. Create MANIFEST.md at the repo root. Leave it empty or add one or two routes for the domains you know you will work in first.
  3. Create .context/ directory. It can be empty. Supply documents will accumulate as the agent pulls the andon cord and you fill the gaps.
  4. Work. When the agent stops and surfaces uncertainty, write the supply document. Add the manifest route. The system builds itself through use.

For an existing project with a large instruction file:

  1. Create the three-tier structure.
  2. Read your existing instruction file line by line. For each line:
    • If it prevents irreversible damage → constitution.
    • If it is reference material for a specific task type → supply document, with a manifest route.
    • If the agent could figure it out from the code → delete it.
  3. Most lines will be supply material or deletable. The constitution will be surprisingly small.
  4. Run the result for a week. The agent’s andon stops will tell you what is missing. Fill the gaps.

For a monorepo or multi-service project:

The standard supports nesting. Each service or domain can have its own .context/ directory with its own supply documents. The root constitution and manifest apply globally. Service-level constitutions (if needed) supplement — never contradict — the root.

<repo-root>/
├── CONSTITUTION.md              ← global hard stops
├── MANIFEST.md                  ← global routing
├── .context/                    ← shared supply
│   └── infrastructure.md
├── services/
│   ├── auth/
│   │   ├── .context/            ← auth-specific supply
│   │   │   └── token-flow.md
│   │   └── src/
│   └── billing/
│       ├── .context/            ← billing-specific supply
│       │   └── payment-rules.md
│       └── src/

What This Standard Does Not Cover

Model selection. Use whatever model works for your task. The standard is model-agnostic.

Agent tooling. Use whatever harness runs your agent. The standard describes repository structure, not tool configuration.

Prompt engineering. The constitution is not a prompt — it is a governance document. How your agent tool loads and presents it is a tool concern, not a standard concern.

Automated context generation. If your tooling can generate repo maps, dependency graphs, or changelogs automatically, great — make them available for pull. The standard only requires that inferable knowledge is system-generated, not human-maintained.

Specific line counts. The budgets in this spec (50 lines for constitution, 15 lines for manifest, 200 lines per supply document) are operational heuristics. They come from practice. Your project may need different thresholds. The principle is: the constitution should be small enough that every line is load-bearing, and the first sign of bloat is the signal to compact.

Summary

The Lean Context standard encodes a cultural contract between a human and an agent into repository structure.

The constitution prevents damage. The manifest enables discovery. The supply provides depth. The andon rule tells the agent when to stop — and the agent helps fill the gap it found. Session discipline keeps the workbench clean: every session ends when the problem is solved or when a gap is discovered and filled. Either way, the repository is better than it was. Lean compaction keeps the system sharp between sessions so the agent never needs to compress within one. The knowledge boundary tells the human where to spend their effort.

The goal is not a well-instructed agent. The goal is a well-structured system that makes the agent — and the human — smarter every session.
